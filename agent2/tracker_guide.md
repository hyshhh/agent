# 目标跟踪原理与参数配置指南

## 整体流程

```
第 N 帧进来
    │
    ▼
┌─────────────┐
│  YOLO 检测   │ → 输出一堆框：[框A(conf=0.9), 框B(conf=0.7), 框C(conf=0.4), 框D(conf=0.2)]
└─────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  第一步：按置信度分三档                                       │
│                                                             │
│  框A(0.9) 框B(0.7)  →  高分框 (conf ≥ track_high_thresh)    │
│  框C(0.4)           →  低分框 (track_low_thresh ≤ conf < high)│
│  框D(0.2)           →  丢弃    (conf < track_low_thresh)     │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  第二步：卡尔曼滤波预测每个人"应该在哪"                        │
│                                                             │
│  假设上一帧有3个轨迹：                                        │
│    track#1: 上一帧在(100,200)，速度向右下                     │
│    track#2: 上一帧在(300,150)，速度向左                       │
│    track#3: 上一帧在(500,300)，静止                          │
│                                                             │
│  卡尔曼预测本帧位置：                                         │
│    track#1 → 预测在(105,205)                                 │
│    track#2 → 预测在(295,150)                                 │
│    track#3 → 预测在(500,300)                                 │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  第三步：高分框 vs 轨迹 匹配（第1轮）                          │
│                                                             │
│  计算每个"高分框"和每个"预测轨迹"的 IoU：                      │
│                                                             │
│              track#1   track#2   track#3                     │
│  框A(0.9)     0.7       0.05      0.01                      │
│  框B(0.7)     0.02      0.6       0.03                      │
│                                                             │
│  匈牙利算法最优匹配：                                          │
│    框A ←→ track#1 (IoU=0.7 > match_thresh ✅ 匹配成功)        │
│    框B ←→ track#2 (IoU=0.6 > match_thresh ✅ 匹配成功)        │
│    track#3 无匹配（本帧没看到）                                │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  第四步：低分框 vs 剩余轨迹 匹配（第2轮）                      │
│                                                             │
│  第1轮没匹配上的轨迹：track#3                                 │
│  第1轮没用上的框：框C(0.4)                                    │
│                                                             │
│  计算 IoU：                                                  │
│    框C vs track#3 → IoU=0.3                                 │
│    0.3 > match_thresh ✅ 匹配成功 → track#3 = 框C             │
│                                                             │
│  这就是低分框的价值：                                          │
│    track#3 的人可能被部分遮挡了，置信度掉到 0.4                 │
│    但通过第二轮匹配，ID 还是保住了                              │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  第五步：处理未匹配的                                         │
│                                                             │
│  未匹配的框 → 创建新轨迹 (track#4, track#5...)                │
│  未匹配的轨迹 → 倒计时 (track_buffer)                         │
│                                                             │
│  track_buffer: 30 意味着：                                   │
│    - 连续30帧没匹配上才真正删除这个 ID                         │
│    - 期间如果重新出现，ID 还是原来的                            │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
输出：track#1=框A, track#2=框B, track#3=框C
```

## 两种跟踪器对比

### ByteTrack

纯 IoU 匹配，不看外观，速度快。

```
匹配代价 = IoU(检测框, 轨迹预测框)
```

- **优点**：参数少，速度快，不会因为穿同样衣服搞混
- **缺点**：人跑远了 IoU=0，就丢 ID

### BoT-SORT

IoU + 运动预测，可选外观特征。

```
with_reid: false  →  匹配代价 = IoU + 运动距离
with_reid: true   →  匹配代价 = α×IoU + β×运动距离 + γ×外观相似度
```

- **优点**：长时间遮挡后仍能找回 ID
- **缺点**：多一个 ReID 模型（with_reid: true 时），吃显存

## 参数详解

### 通用参数（bytetrack / botsort 共用）

#### `track_high_thresh` — 高分框门槛

```
conf ≥ track_high_thresh 的框 → 参与第1轮匹配
```

- 提高（如 0.6）→ 第1轮更准，减少误匹配，但半遮挡的人可能进不了第1轮
- 降低（如 0.3）→ 第1轮框多，更容易匹配，但噪声也更多

#### `track_low_thresh` — 低分框门槛

```
track_low_thresh ≤ conf < track_high_thresh 的框 → 参与第2轮匹配
conf < track_low_thresh → 直接丢弃
```

- 提高（如 0.3）→ 只有较高置信度的低分框才能参与二次匹配，减少噪声
- 降低（如 0.1）→ 更多低分框参与，遮挡恢复能力更强，但噪声也更多

**两阶段匹配的意义：**

```
1.0 ┤ ██
0.9 ┤ ████
0.8 ┤ ██████
0.7 ┤ ████████        ← 高分框区域 (≥track_high_thresh)
0.6 ┤ ██████████      ← track_high_thresh
    │ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─
0.5 ┤ ████████████
0.4 ┤ ██████████████  ← 低分框区域 (track_low_thresh ~ track_high_thresh)
0.3 ┤ ████████████████← track_low_thresh
    │ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─
0.2 ┤ ██████████████████ ← 丢弃区域 (<track_low_thresh)
0.1 ┤ ████████████████████
```

#### `match_thresh` — 匹配阈值

```
IoU > match_thresh → 认为是同一个人 → 匹配成功
```

| 值 | 效果 | 风险 |
|----|------|------|
| 0.01~0.05 | 极宽松 | 相邻两人被合并成同一 ID |
| 0.3~0.5 | 推荐范围 | 平衡 |
| 0.8+ | 极严格 | 框稍微偏移就换 ID |

**示例：**

```
场景：两个人站得比较近

    人A的框               人B的框
  ┌──────────┐         ┌──────────┐
  │          │         │          │
  │          │    重叠  │          │
  │          │◀─区域──▶│          │
  └──────────┘         └──────────┘
         └───IoU≈0.15───┘

match_thresh=0.05:  0.15 > 0.05 ✅ → 被当成同一个人（错误！）
match_thresh=0.4:   0.15 < 0.4  ❌ → 正确识别为两个人
```

#### `track_buffer` — 跟踪丢失容忍帧数

```
连续 track_buffer 帧没匹配上 → 彻底删除该 ID
```

```
帧1: track#1 看到人A ✅
帧2: track#1 看到人A ✅
帧3: 人A被柱子遮挡 ❌ → track#1 倒计时: 29
帧4: 还没看到        → track#1: 28
...
帧32: 还没看到       → track#1: 1
帧33: 还没看到       → track#1 彻底删除
帧34: 人A重新出现    → 新建 track#5（旧ID已释放）
```

| 值 | 30fps 下容忍时间 | 适用场景 |
|----|-----------------|---------|
| 5 | 0.17 秒 | 几乎不允许遮挡 |
| 15 | 0.5 秒 | 轻度遮挡 |
| 30 | 1 秒 | 推荐值 |
| 60 | 2 秒 | 长时间遮挡场景 |

### BoT-SORT 专有参数

#### `with_reid` — 是否启用 ReID 外观特征

| 值 | 效果 | 代价 |
|----|------|------|
| `false` | 纯运动 + IoU 匹配 | 无额外开销 |
| `true` | 加入外观相似度匹配 | 需下载 ReID 模型，额外 GPU 显存 |

**什么时候开 ReID：**
- 画面中有多人频繁交叉
- 人被长时间完全遮挡后再出现
- ID 稳定性比速度更重要

## 参数之间的依赖关系

```
detector.confidence ──→ 必须 ≤ tracker.track_low_thresh

  如果 confidence > track_low_thresh：
    → 低分框根本没进入 tracker，第二阶段匹配形同虚设

  例：confidence=0.4, track_low_thresh=0.3
    → YOLO 只输出 conf≥0.4 的框
    → tracker 要找 conf 在 0.3~0.4 之间的低分框 → 找不到
    → 第二阶段失效

detector.nms_iou ──→ YOLO 内部 NMS，控制合并重叠框的严格程度

  太小（0.1）→ 不同人的框被合并 → 丢人
  太大（0.7）→ 同一个人保留多个框 → tracker 给每个框分配独立 ID

tracker.match_thresh ──→ 框与轨迹的 IoU 匹配门槛

  太小 → 不同人被合并成同一 ID
  太大 → 频繁换 ID

tracker.track_buffer ──→ 丢帧容忍度

  太小 → 遮挡后换 ID
  太大 → 真换人后旧 ID 久不释放
```

## 推荐配置

### 高精度场景（监控、行为分析）

```yaml
detector:
  confidence: 0.3
  nms_iou: 0.3
tracker:
  enabled: true
  tracker_type: "bytetrack"
  track_high_thresh: 0.6
  track_low_thresh: 0.3
  match_thresh: 0.4
  track_buffer: 30
```

### 高速场景（实时性优先）

```yaml
detector:
  confidence: 0.5
  nms_iou: 0.5
tracker:
  enabled: true
  tracker_type: "bytetrack"
  track_high_thresh: 0.5
  track_low_thresh: 0.2
  match_thresh: 0.5
  track_buffer: 15
```

### 需要长时间稳定 ID（多人交叉场景）

```yaml
detector:
  confidence: 0.3
  nms_iou: 0.4
tracker:
  enabled: true
  tracker_type: "botsort"
  track_high_thresh: 0.6
  track_low_thresh: 0.3
  match_thresh: 0.4
  track_buffer: 50
  with_reid: true
```

### 仅检测不需要跟踪

```yaml
tracker:
  enabled: false
```
